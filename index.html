Basic Java Core Foundation

1. Java Fundamentals

Java is a class-based, object-oriented programming language designed for platform independence via the JVM.

History of Java

Created by James Gosling at Sun Microsystems in 1995. Originally called "Oak" (after a tree outside Gosling's window). Designed with the principle "Write Once, Run Anywhere" (WORA). Sun was acquired by Oracle in 2010. Major releases: Java 8 (2014, lambdas), Java 11 (2018, LTS), Java 17 (2021, LTS), Java 21 (2023, LTS).

Major Java Milestones

Java 8 (2014): Lambdas, Streams, Optional, Date/Time API. Java 9 (2017): Modules (JPMS), JShell, private interface methods. Java 11 (2018 LTS): HTTP Client, var for lambdas, flight recorder. Java 17 (2021 LTS): Sealed classes, pattern matching, records. Java 21 (2023 LTS): Virtual threads, record patterns, structured concurrency.

Features of Java

  • Platform Independent: Bytecode runs on any JVM.
  • Object-Oriented: Everything (except primitives) is an object.
  • Automatic Memory Management: Garbage Collector handles cleanup.
  • Robust & Secure: No pointers, bytecode verification, sandboxing.
  • Multithreaded: Built-in thread support.
  • Rich Standard Library: Collections, I/O, networking, concurrency.
Why Java Dominates Enterprise

Backward compatibility (20+ year-old code runs on modern JVMs), massive ecosystem (Spring, Hibernate, Maven), strong typing catches bugs early, and the JVM is a world-class runtime with advanced JIT compilation and GC algorithms.

Java Program Structure

Every Java program is a class with a main method — the entry point. The file name must match the public class name.

💡 Interview Tip

Common question: "Why is Java platform-independent?"

Answer: Java code is compiled to bytecode (platform-independent) by javac. The JVM then interprets/JIT-compiles this bytecode to platform-specific machine code. The same .class file runs on any platform that has a JVM implementation.

Follow-up: "What's the difference between JVM, JRE, and JDK?" — Be ready with the nesting diagram: JDK ⊃ JRE ⊃ JVM.

Basic Java program structure
// HelloWorld.java — filename matches class name
public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello, Java!");
    }
}

// Compile: javac HelloWorld.java  → produces HelloWorld.class
// Run:     java HelloWorld         → JVM loads, executes main()
Key: Java = compiled to bytecode (javac) → interpreted by JVM (java). WORA.

Memory Footprint Comparison

boolean: JVM-dependent (~1 byte). byte: 1 byte. short/char: 2 bytes. int/float: 4 bytes. long/double: 8 bytes. An int[] of 10M elements uses ~40MB. Choosing the right type matters in big data.

Type Casting

  • Widening (automatic): Smaller type → larger type. int → long → double. No data loss. Done implicitly.
  • Narrowing (explicit): Larger type → smaller type. double → int. Data loss possible. Must cast: (int) 3.14 → 3.
Numeric Promotion Rule

In expressions, byte, short, char are promoted to int automatically. byte a = 10; byte b = 20; byte c = a + b; — COMPILE ERROR, a + b is int. Must cast: byte c = (byte)(a + b);

💡 Interview Tip

Common question: "What's the default value of a boolean? What about int?"

  • boolean: false
  • int, byte, short, long, float, double: 0
  • char: '\u0000' (null character)
  • Object references: null

Trick question: "Can you assign a double to an int?" — Only with explicit narrowing cast: (int) 3.14. Without the cast, the compiler will refuse.

Type casting examples
// Widening — automatic, safe
int i = 100;
long l = i;        // int → long, no cast needed
double d = i;      // int → double, no cast needed

// Narrowing — explicit, risk of data loss
double pi = 3.14159;
int truncated = (int) pi;  // 3 — decimal part lost!

long big = 1_000_000_000_000L;
int small = (int) big;     // Overflow! Data corrupted
Key: 8 primitives (byte, short, int, long, float, double, char, boolean). Widening = automatic. Narrowing = explicit cast needed.

3. Variables

A variable is a named memory location that stores a value. Java has four types based on scope and behavior.

1. Local Variables

Declared inside a method. Must be initialized before use (no default value). Stored on the Stack. Destroyed when the method ends.

Stack vs Heap

Local variables and parameters live on the Stack (per-thread). Objects live on the Heap (shared). The reference variable is on the Stack — the object itself is on the Heap.

2. Instance Variables (Non-Static Fields)

Declared inside a class but outside any method, without static. Each object has its own copy. Stored on the Heap. Get default values (0, null, false).

Default Values Guarantee

Instance variables get default values (0, false, null) before the constructor runs. This is guaranteed by the JVM. Local variables have NO default — the compiler catches uninitialized usage.

3. Static Variables (Class Variables)

Declared with static. One copy shared across all instances. Stored in Metaspace. Initialized when the class is loaded.

Static Variable Internals

Stored in Metaspace. Initialized when class is loaded. final static fields get thread-safe initialization guarantee from JLS.

4. Final Variables (Constants)

Declared with final. Can't be reassigned after initialization. For primitives: the value is fixed. For references: the reference is fixed (but the object's state can change).

💡 Interview Tip

Common question: "What's the difference between instance variables and static variables?"

Instance variables = one per object (Heap). Static variables = one per class (Metaspace). Instance variables are created when new is called. Static variables are created when the class is loaded.

Follow-up: "Can a static method access an instance variable?" — No. Static methods don't have a this reference. They can only access static variables or local variables.

Variable types in action
public class Car {
    static int totalCars;     // Static — shared, default 0
    String model;             // Instance — per object, default null
    final int WHEELS = 4;     // Final — constant, can't change

    void drive(int speed) {   // speed = parameter (local-like)
        int distance = 0;     // Local — must initialize
        distance = speed * 10;
        System.out.println(model + " drove " + distance + "km");
    }
}
Key: Static = class level (Metaspace). Instance = per object (Heap). Local = per method (Stack). Final = read-only.

4. Operators

Operators perform operations on operands. Java has 7 categories.

Arithmetic: + - * / %
Relational: == != < > <= >=
Logical: && || !
Bitwise: & | ^ ~ << >> >>>
Assignment: = += -= *= /= %=
Unary: ++ -- ! ~
Ternary: condition ? valueIfTrue : valueIfFalse

Key Nuances

  • && and || are short-circuit: if the first operand determines the result, the second is not evaluated.
  • & and | are non-short-circuit: both sides always evaluated.
  • i++ (post-increment): use then increment. ++i (pre-increment): increment then use.
  • >>> (unsigned right shift): shifts zeros in from the left (positive result for negative numbers).
💡 Interview Tip

Common question: "What's the difference between && and &?"

&& is short-circuit — if the left side is false, the right side is never evaluated. & always evaluates both sides. Same for || vs |.

Example: if (str != null && str.length() > 0) — safe with &&, would throw NPE with &.

Short-circuit vs non-short-circuit
int x = 5, y = 0;

// Short-circuit && — y != 0 is NOT evaluated (safe)
if (x > 0 && (y / x) > 0) { }

// Non-short-circuit & — both sides evaluated (risk!)
if (x > 0 & (y / x) > 0) { }  // Division by zero! 😱
Key: &&/|| stop early. &/| don't. ++i vs i++ matters. Ternary is elegant for simple if-else.

5. Control Statements

Control flow statements determine the order in which code executes — decision-making, looping, and branching.

Decision-Making

  • if / if-else: Execute block based on boolean condition.
  • switch: Multi-way branch based on a value. Java 14+ supports switch expressions with -> syntax (no fall-through) and yield to return values.
Ternary Operator Type Inference

Object result = condition ? 42 : "text" — return type is Object (common supertype). Only one branch evaluates (no side effects from unselected branch).

Looping

  • for: When you know the iteration count. for (init; condition; update)
  • while: When you don't know the count. Checks condition first.
  • do-while: Executes at least once (condition checked after body).
Enhanced For-Loop Internals

For arrays: compiled to standard indexed for. For Iterable: compiled to Iterator usage. The iterator's hasNext()/next() is called each iteration. Concurrent modification during iteration throws ConcurrentModificationException.

Branching

  • break: Exit the loop immediately.
  • continue: Skip to the next iteration.
  • return: Exit the method (with optional value).
  • labeled break/continue: Break out of nested loops. outer: for(...) { ... break outer; }
💡 Interview Tip

Common question: "What's new in switch expressions (Java 14+)?"

The new switch is an expression (returns a value), uses -> (no fall-through), supports multi-case labels (case MONDAY, FRIDAY), and uses yield for block bodies. No more break statements needed.

Question: "When would you use a labeled break?" — To break out of outer loops from nested loops: outer: for(...) { for(...) { break outer; } }

Switch expression (Java 14+)
// Old-style switch (fall-through risk)
String oldResult;
switch (day) {
    case MONDAY:
    case FRIDAY: oldResult = "Work"; break;
    default: oldResult = "Unknown";
}

// Modern switch expression (no fall-through, returns value)
String result = switch (day) {
    case MONDAY, FRIDAY -> "Work";
    case SATURDAY, SUNDAY -> "Weekend";
    default -> "Unknown";
};
Key: Modern switch = no fall-through, returns values with -> or yield.

6. Arrays

Arrays are objects that hold a fixed number of values of a single type. Indexed, 0-based, contiguous memory.

Single-Dimensional Array

A linear sequence of elements. int[] numbers = new int[5]; Creates 5 slots, all default 0. Access: numbers[0].

Array Initialization Syntaxes

int[] a = new int[5]; — defaults to 0. int[] a = {1, 2, 3}; — shorthand (declaration only). int[] a = new int[]{1, 2, 3}; — anonymous (anywhere).

Multi-Dimensional Array

Array of arrays (not a matrix). int[][] matrix = new int[3][4]; — 3 rows, 4 columns. In Java, rows can have different lengths (jagged arrays).

Important: Arrays Are Objects

Arrays have .length (not a method — a field!). They inherit from Object. They can be passed by reference. They are covariant: String[] is a subtype of Object[] (which can cause ArrayStoreException).

Array Covariance Problem

String[] s = new String[10]; Object[] o = s; o[0] = 42; — ArrayStoreException at runtime. Arrays are covariant (unlike invariant generics).

💡 Interview Tip

Common question: "How do you find the length of an array?"

Arrays use the .length field (not a method). Strings use .length() method. This is a classic Java gotcha.

Follow-up: "What is ArrayStoreException?" — Arrays are covariant (String[] is a subtype of Object[]). Storing a non-String into a String[] throws ArrayStoreException at runtime.

Array operations
int[] nums = {10, 20, 30, 40, 50};
System.out.println(nums.length);   // 5
nums[2] = 35;                       // Overwrite

// Multi-dimensional (jagged)
int[][] matrix = {
    {1, 2, 3},
    {4, 5},
    {6, 7, 8, 9}
};
System.out.println(matrix[0].length); // 3
System.out.println(matrix[1].length); // 2

// Array covariant — can cause runtime exception
Object[] objs = new String[3];
objs[0] = "hello";         // OK
objs[1] = 42;              // ArrayStoreException at runtime!
Key: Fixed size, 0-indexed, .length field, covariant (runtime check).

7. Methods

A method is a block of code that performs a specific task. It's the fundamental building block of behavior in Java.

Method Declaration

A method has: access modifier → return type → name → parameters → body. public int add(int a, int b) { return a + b; }

Method Signature

The method signature is the name + parameter types (return type is NOT part of the signature). This is why overloading cannot differ by return type alone — the compiler wouldn't know which overload to call.

Method Overloading

Multiple methods with the same name but different parameters (count, type, or order). Compile-time polymorphism. Return type alone is not enough for overloading — the parameter list must differ.

Overloading Resolution Order

Exact match → Widening → Autoboxing → Varargs. Example: print(5) with print(int), print(long), print(Integer), print(int...) picks print(int).

Pass-by-Value

Java is strictly pass-by-value. For primitives, the value is copied. For objects, the reference is copied (but it's still pass-by-value). You can't swap objects in a method — the caller retains the original reference.

The Swap Method Fallacy

void swap(Object a, Object b) { Object t = a; a = b; b = t; } — does NOT swap caller's variables. Java copies reference values. Use array or mutable container to swap.

💡 Interview Tip

Common question: "Can you overload a method by changing the return type only?"

No. Overloading requires different parameter lists (count, type, or order). Return type alone is not sufficient — the compiler can't distinguish based on return type alone.

Follow-up: "Is Java pass-by-value or pass-by-reference?" — Always pass-by-value. For primitives, the value is copied. For objects, the reference value is copied (not the object). You cannot swap objects in a method.

Method overloading
public class Calculator {
    // Same name, different parameters
    public int add(int a, int b) { return a + b; }
    public double add(double a, double b) { return a + b; }
    public int add(int a, int b, int c) { return a + b + c; }

    // NOT valid overloading — only return type differs!
    // public long add(int a, int b) { return a + b; }
}

// Java is pass-by-value:
void swap(String a, String b) {
    String temp = a; a = b; b = temp;  // only swaps local copies
}
String x = "A", y = "B";
swap(x, y);
// x still "A", y still "B" — no swap!
Key: Overloading = same name, different params (compile-time). Java = pass-by-value (always).

8. String

String is immutable, stored in the String Pool (Heap), and one of the most important classes in Java.

String — Immutable

Once created, a String cannot be changed. Every "modification" creates a new String object. This is for security, caching, and thread safety.

Why Immutability Wins

Thread-safe: No sync needed. Hash caching: Computed once. Security: Safe for class loading. String Pool: Enables interning across JVM.

StringBuilder vs StringBuffer

  • StringBuilder: Mutable, not thread-safe, faster. Use in single-threaded code.
  • StringBuffer: Mutable, thread-safe (synchronized methods), slower. Use in multithreaded code.
  • Both grow dynamically — no new object per modification.

Key String Methods

length(), charAt(), substring(), indexOf(), equals(), equalsIgnoreCase(), toLowerCase(), toUpperCase(), trim(), replace(), split(), join(), intern().

String Pool

Literals ("hello") are interned — the JVM caches them. new String("hello") creates a new object on the heap (not in pool). .intern() returns the pooled reference.

💡 Interview Tip

Common question: "Why is String immutable?"

1) Security: String parameters are used in class loading, networking, file paths — mutable strings could bypass checks. 2) String Pool: Immutability makes caching possible. 3) Thread safety: No synchronization needed. 4) HashCode caching: HashMap keys can be Strings because the hash is computed once.

Follow-up: "StringBuilder vs StringBuffer?" — StringBuilder is not thread-safe (but faster). StringBuffer is synchronized (but slower). Use StringBuilder in single-threaded code.

String, StringBuilder, StringBuffer
// String — immutable, each += creates new object
String s = "a";
s += "b";  // creates a NEW String "ab", old "a" is GC'd

// StringBuilder — mutable, no new objects
StringBuilder sb = new StringBuilder("a");
sb.append("b");  // modifies the same object

// String comparison
String s1 = "hello";
String s2 = "hello";
String s3 = new String("hello");

System.out.println(s1 == s2);       // true (same pool reference)
System.out.println(s1 == s3);       // false (different objects)
System.out.println(s1.equals(s3));  // true (same value)
Key: String = immutable, pooled. StringBuilder = mutable, fast (single-thread). Use .equals() not == for value comparison.

9. Object-Oriented Programming

The four pillars of OOP — classes, objects, and the relationships between them.

Classes & Objects

A class is a blueprint. An object is an instance of a class. Car myCar = new Car(); — Car is the class, myCar is the object.

Object Lifecycle

Declaration → new (Heap allocation) → Constructor → Usage → Unreachable → GC. The new keyword allocates, runs constructor, returns reference.

Constructor

A special method that runs when an object is created. Same name as the class, no return type. If you don't define one, the compiler provides a default no-arg constructor. this() calls another constructor in the same class. super() calls the parent constructor.

Constructor Chaining

First statement: super() or this(). If omitted, compiler inserts super(). If superclass has no default constructor, call super(args) explicitly.

Encapsulation

Hide internal state with private fields, expose via public getters/setters. This gives you control to add validation, change internal representation, or trigger side effects.

Defensive Copying

Never return mutable internal fields: public Date getBirthday() { return new Date(birthday.getTime()); }. Immutable fields (final LocalDate) eliminate the problem.

Inheritance

  • Single: Class A extends Class B (one parent only).
  • Multilevel: A extends B extends C (chain).
  • Hierarchical: B extends A, C extends A (siblings).
  • No multiple inheritance of classes (use interfaces instead).
Liskov Substitution Principle

Subtypes must be substitutable for base types. Violations: throwing new exceptions, weakening visibility, strengthening preconditions. Classic: Square extends Rectangle breaks area behavior.

Polymorphism

  • Compile-time (Overloading): Same method name, different parameters.
  • Runtime (Overriding): Child class redefines a parent method. The actual method called is determined at runtime based on the object's type.
Virtual Method Dispatch

Non-static, non-final methods are virtual. JVM uses vtable with invokevirtual bytecode. Fields are NOT polymorphic (compile-time binding).

Abstraction

Hide implementation details. Abstract classes can have both abstract (no body) and concrete methods. Cannot be instantiated. Subclasses must implement abstract methods.

💡 Interview Tip

Common question: "Explain all four pillars of OOP."

1) Encapsulation: Bundle data + methods, hide internals with private/getters/setters. 2) Inheritance: Code reuse via extends. 3) Polymorphism: Same interface, different implementations (overloading = compile time, overriding = runtime). 4) Abstraction: Hide complexity, show only essentials (abstract classes, interfaces).

Follow-up: "Difference between abstract class and interface?" — Abstract class: can have state (fields), can have constructors, single inheritance. Interface: no state (until Java 8), multiple inheritance, implements keyword.

Polymorphism — runtime example
abstract class Animal {
    abstract void speak();
}

class Dog extends Animal {
    void speak() { System.out.println("Bark!"); }
}

class Cat extends Animal {
    void speak() { System.out.println("Meow!"); }
}

// Runtime polymorphism:
Animal a = new Dog();
a.speak();  // "Bark!" — JVM checks actual type at runtime
a = new Cat();
a.speak();  // "Meow!"
Key: 4 pillars: Encapsulation, Inheritance, Polymorphism, Abstraction. Overriding = runtime. Overloading = compile time.

10. Packages

Packages organize classes into namespaces — preventing naming conflicts and controlling access.

Built-in Packages

java.lang (auto-imported: String, Math, System), java.util (collections, Scanner), java.io (I/O streams), java.net (networking), java.nio (NIO), java.time (Java 8 date/time).

Essential Java Packages

User-Defined Packages

12. Object Class Methods

java.lang.Object is the root of the Java class hierarchy — every class inherits its 11 methods. The contract of each is critical for interviews.

equals() — Value Equality

The default implementation in Object uses == (reference equality). Override it to provide logical equality. The contract (from Java docs): reflexive (x.equals(x) true), symmetric (x.equals(y) == y.equals(x)), transitive, consistent (same result if unchanged), non-null (x.equals(null) false).

equals() Implementation Template

@Override public boolean equals(Object o) { if (this == o) return true; if (o == null || getClass() != o.getClass()) return false; Person p = (Person) o; return age == p.age && name.equals(p.name); }. Use getClass() check (not instanceof) for symmetric equals. Always override hashCode() when overriding equals().

hashCode() — Hash Contract

The golden rule: If a.equals(b) then a.hashCode() == b.hashCode(). The reverse is NOT required (collisions are allowed). Never use hashCode() for equality checks. Default Object.hashCode() is typically the memory address (not guaranteed, JVM-specific).

hashCode() Implementation

@Override public int hashCode() { return Objects.hash(name, age); } — Java 7+ Objects.hash() delegates to Arrays.hashCode(). For performance: int result = name.hashCode(); result = 31 * result + age; return result;. The prime 31 reduces collisions (odd prime, good distribution).

toString() — String Representation

Default: ClassName@hashCode (e.g., Person@1a2b3c). Override to provide meaningful debug info. @Override public String toString() { return "Person{name='" + name + "', age=" + age + "}"; }. Records auto-generate toString().

clone() — Object Copying

Must implement Cloneable marker interface, otherwise throws CloneNotSupportedException. Protected method in Object. The default super.clone() performs a shallow copy — reference fields are still shared. Deep copy: override clone() and manually clone mutable fields.

Shallow vs Deep Copy

Shallow: super.clone() — the new object's reference fields point to the same objects. Deep: clone() manually copies mutable fields (e.g., public class Person implements Cloneable { int[] scores; @Override public Person clone() throws CloneNotSupportedException { Person p = (Person) super.clone(); p.scores = scores.clone(); return p; } }).

getClass() — Runtime Type

final method — returns the Class<?> object representing the runtime class. obj.getClass() == Person.class checks exact type. obj instanceof Person checks type or subtype. getClass() is used for type-safe equals implementations.

finalize() — Deprecated Cleanup

Called by GC before reclaiming the object. Deprecated since Java 9. Unreliable (may never run), performance-killing (delays GC). Modern replacements: AutoCloseable + try-with-resources, Cleaner (Java 9+), PhantomReference.

Interview Tip

Most asked question: "Explain the equals() and hashCode() contract." — Always start with: "If equals returns true, hashCodes must be equal. But equal hashCodes doesn't mean equals is true (collisions)."

Follow-up: "What happens if you put an object in a HashMap, then change a field used in hashCode()?" — The object is in the wrong bucket. You can't find it with get(). The entry becomes orphaned. This is why keys in HashMap should be immutable.

Anti-pattern: Using instanceof in equals() breaks symmetry: class A { @Override equals(Object o) { return o instanceof A; } } class B extends A { @Override equals(Object o) { return o instanceof B; } } — new A().equals(new B()) is true, but new B().equals(new A()) is false. Always use getClass() for symmetric equals.

Object class methods — equals, hashCode, toString, clone
public class Employee implements Cloneable {
    private final int id;
    private String name;
    private Department dept;  // mutable field

    public Employee(int id, String name, Department dept) {
        this.id = id;
        this.name = name;
        this.dept = dept;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Employee e = (Employee) o;
        return id == e.id && Objects.equals(name, e.name);
    }

    @Override
    public int hashCode() {
        return Objects.hash(id, name);
    }

    @Override
    public String toString() {
        return "Employee{id=" + id + ", name='" + name + "'}";
    }

    @Override
    public Employee clone() throws CloneNotSupportedException {
        Employee e = (Employee) super.clone();
        e.dept = this.dept.clone();  // deep copy mutable field
        return e;
    }
}

Key: equals + hashCode = must be overridden together. Immutable keys in HashMap. Shallow copy from super.clone(), deep copy mutable fields. toString() for debug.

13. BigDecimal & Math API

Floating-point arithmetic with double/float is imprecise. BigDecimal provides exact decimal arithmetic — critical for financial applications.

Why Not double for Money

0.1 + 0.2 = 0.30000000000000004 (IEEE 754 binary floating-point can't represent 0.1 exactly). For money, use BigDecimal with String constructor: new BigDecimal("0.10"). Never use new BigDecimal(0.1) (still imprecise — uses double literal).

BigDecimal vs BigInteger vs AtomicInteger

BigDecimal: Arbitrary-precision decimal numbers (money, exact calculations). BigInteger: Arbitrary-precision integers (cryptography, very large numbers). AtomicInteger: Thread-safe int for concurrent counters. Each serves a different purpose.

BigDecimal Scale and Precision

Scale = number of digits after decimal. new BigDecimal("10.50") has scale 2. Precision = total digits. setScale(2, RoundingMode.HALF_UP) for money. Always specify RoundingMode — the default is UNNECESSARY (throws ArithmeticException if rounding needed).

Rounding Modes

HALF_UP: Standard rounding (4 → down, 5 → up). HALF_EVEN: Banker's rounding (round to even when exactly 0.5) — reduces bias over many operations. UP/DOWN: Always away from zero / toward zero. CEILING/FLOOR: Toward positive/negative infinity. Use HALF_UP for display, HALF_EVEN for statistical calculations.

Common Pitfalls

  • equals() vs compareTo(): new BigDecimal("2.0").equals(new BigDecimal("2.00")) → false (different scale). compareTo() returns 0 (same value). Use compareTo() for value comparison.
  • Immutable: Every operation returns a new BigDecimal (like String). Don't forget to assign the result.
  • Performance: BigDecimal is 10-50x slower than double. Not for high-frequency trading tick data.

Math API Essentials

Math.min/max, Math.abs, Math.round/ceil/floor, Math.pow/sqrt, Math.random, Math.sin/cos/tan. Also: Math.floorDiv() (floor division, correct for negatives), Math.floorMod() (floor modulus). Math.addExact() / multiplyExact() (throw ArithmeticException on overflow). StrictMath for platform-independent results.

Interview Tip

Common question: "How would you represent money in Java?" — Always BigDecimal with String constructor. Use setScale(2, RoundingMode.HALF_UP). Use compareTo() for comparison, not equals().

Follow-up: "What's the difference between HALF_UP and HALF_EVEN?" — HALF_EVEN (banker's rounding) rounds 2.5 to 2, 3.5 to 4. Over many operations, the bias is zero. HALF_UP always rounds 0.5 up — builds in a systematic positive bias.

BigDecimal best practices for money
// WRONG — imprecise
double price = 0.1 + 0.2;  // 0.30000000000000004

// RIGHT — exact
BigDecimal price = new BigDecimal("0.10")
    .add(new BigDecimal("0.20"));  // 0.30

// Always set scale and rounding mode
BigDecimal total = new BigDecimal("99.99")
    .multiply(new BigDecimal("1.08"))    // 8% tax
    .setScale(2, RoundingMode.HALF_UP);

// Comparison — use compareTo, not equals!
BigDecimal a = new BigDecimal("2.0");
BigDecimal b = new BigDecimal("2.00");
a.equals(b);      // false — different scale!
a.compareTo(b);   // 0 — same value!

// Overflow-safe math
int safe = Math.addExact(Integer.MAX_VALUE - 1, 1);  // OK
int boom = Math.addExact(Integer.MAX_VALUE, 1);       // ArithmeticException!
Key: BigDecimal for money. String constructor. Always set scale + rounding. compareTo() for comparison. Immutable — assign result.

14. Logging in Java

Logging is the primary observability tool for production Java applications. Slf4J is the industry-standard facade.

Logging Facades vs Implementations

Facade (API): SLF4J, Apache Commons Logging. Your code depends on these — they define the API. Implementation: Logback, Log4j2, java.util.logging. At runtime, the facade delegates to the implementation. SLF4J + Logback is the most popular combination.

Why Use a Facade?

Your code should never depend directly on Logback or Log4j2. If you switch implementations, you change only the classpath, not your code. SLF4J also supports parameterized logging: log.info("User {} logged in from {}", user, ip) — avoids string concatenation when the log level is disabled.

Log Levels (Standard)

ERROR: Something is broken (the app can't continue). WARN: Something unexpected happened (but the app continues). INFO: Important business events (user login, order placed). DEBUG: Development-time details (method entry/exit, SQL parameters). TRACE: Very fine-grained (step-by-step execution). In production, usually set to INFO or WARN.

Logback Configuration (logback.xml)

Configure appenders (where logs go) and loggers (which packages at which level). Common appenders: ConsoleAppender (stdout), RollingFileAppender (daily or size-based rotation), SyslogAppender. Pattern layout: %d{HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n.

Best Practices

  • Parameterized logging: log.info("Order {} processed in {}ms", orderId, duration); — faster than + concatenation when level is disabled.
  • Avoid logging in hot loops: Logging is I/O-bound. Guard with if (log.isDebugEnabled()) if the log message is expensive to construct.
  • Log context: Use MDC (Mapped Diagnostic Context) for request-scoped data: MDC.put("requestId", uuid) — included in every log line.
  • Never log sensitive data: No passwords, credit card numbers, PII. Use a whitelist approach.
  • Structured logging: {"timestamp": "...", "level": "INFO", "message": "...", "userId": 123} — JSON format for log aggregation tools (ELK, Splunk, Datadog).
Interview Tip

Common question: "Why SLF4J instead of direct Logback?" — SLF4J is a facade. Your code doesn't depend on Logback at compile time. If you deploy in an environment that uses Log4j2, you just swap the binding JAR. No code changes.

Follow-up: "What's MDC?" — Mapped Diagnostic Context is a thread-local map. Set once per request (MDC.put("requestId", id)), included in all subsequent log lines. Enables tracing requests across microservices.

IS-A Relationship (Inheritance)

A Dog IS-A Animal. Expressed with extends (class) or implements (interface). Represents a hierarchical, "type of" relationship.

Favor Composition Over Inheritance

Inheritance breaks encapsulation (subclass depends on parent). Composition: class Car { Engine e; List w; } not class Car extends Engine. Testable, flexible.

HAS-A Relationship (Association)

A Car HAS-A Engine. Expressed with fields. Represents ownership or usage. Three types:

  • Association: A doctor uses a stethoscope. Loose, independent lifecycle.
  • Aggregation: A team has players. Players can exist without the team (weaker ownership). The child can outlive the parent.

Aggregation vs Composition

Common question: "What package is automatically imported in every Java file?"

💡 Interview Tip

Common question: "Can a top-level class be private or protected?"

No. Top-level classes can only be public or default (package-private). private and protected only apply to inner classes.

Follow-up: "What's the access level of protected?" — Same package + subclasses (even in different packages). The table: private → default → protected → public.

💡 Interview Tip

Common question: "What's the difference between Association, Aggregation, and Composition?"

Association: Loose "uses-a" relationship. Both objects have independent lifecycles. Aggregation: "Has-a" with weak ownership. Child can exist without parent (Team → Player). Composition: "Has-a" with strong ownership. Child dies with parent (House → Room).

UML notation: Association = simple line, Aggregation = empty diamond, Composition = filled diamond.

A contract that defines what a class must do, but not how. Since Java 8, interfaces can also provide default behavior.

Interface Basics

An interface declares abstract methods (no body). A class implements an interface and must provide implementations for all abstract methods. A class can implement multiple interfaces — Java's answer to multiple inheritance.

Interface vs Abstract Class — Quick Reference

Interface: implements, no state (except static final), multiple inheritance, all methods public. Abstract class: extends, can have state and constructors, single inheritance, can have protected/private methods.

Interface Evolution (Java 8+)

  • Default methods (default void foo() {}): Provide a default implementation. Subclasses can override or keep the default. Solves the "add method to interface without breaking all implementations" problem.
  • Static methods (static void bar() {}): Utility methods that belong to the interface, not to instances. Called as MyInterface.bar().
  • Private methods (Java 9+): Helper methods shared among default/static methods within the interface. Not accessible outside.

Functional Interface

An interface with exactly one abstract method. Can be annotated with @FunctionalInterface (compiler enforces it). The foundation of lambda expressions: Runnable, Callable, Comparator, and all java.util.function interfaces.

@FunctionalInterface

Exactly one abstract method. Compiler enforces it. Examples: Runnable, Callable, Comparator, all java.util.function.*. Default methods don't count toward SAM count.

Marker Interface

An interface with no methods — it just marks a class as having a certain capability. Examples: Serializable, Cloneable, RandomAccess. The JVM or library checks instanceof Serializable rather than calling a method.

Common question: "Can you have multiple inheritance in Java?"

3. Enums

A special type that defines a fixed set of constants. Far more powerful than public static final ints.

Enum Basics

enum Color { RED, GREEN, BLUE } — each constant is an instance of the enum type. Type-safe (you can't pass an int where an enum is expected).

Enum Safety Guarantees

Constructors are private. Cannot new an enum. Reflection-proof. Serialization-safe. Inherently thread-safe (constants created once at class init).

Enum Constructors, Fields, Methods

Enums can have constructors (only private), fields, and methods. Each constant can have its own behavior by overriding a method.

Built-in Enum Methods

  • values() — returns all constants as an array.
  • valueOf(String) — returns the constant with that name.
  • ordinal() — returns the position (0-based).
  • name() — returns the constant name as a string.
💡 Interview Tip

Common question: "Can an enum have a constructor?"

Yes, but it must be private (or default). Enum constants are created at class initialization time, not by users. Enums can have fields, methods, and even implement interfaces.

Key point: Enum constants are the only way to create instances — you cannot new an enum. This makes enums inherently singleton per constant.

Enum with custom fields and behavior

    private final String symbol;

Caching

Wrapper classes cache common values: Integer caches -128 to 127. Boolean caches TRUE/FALSE. Byte, Short, Long cache -128 to 127. Character caches 0–127. This is why Integer.valueOf(100) == Integer.valueOf(100) is true but Integer.valueOf(200) == Integer.valueOf(200) is false.

Integer Cache Configuration

💡 Interview Tip

Common question: "Will Integer.valueOf(100) == Integer.valueOf(100) be true?"

Yes for 100, no for 200. Integer caches values from -128 to 127. valueOf() returns cached objects in that range. Outside it, new objects are created. Always use .equals() for wrapper comparison.

Mnemonic: The == operator compares references, not values. .equals() compares values.

7. Generics

Enable types (classes/interfaces) to be parameterized by other types. Provides compile-time type safety — catch type errors at compile time, not runtime.

Generic Classes & Methods

public class Box<T> { T item; void set(T item) { ... } T get() { ... } }. T is a type parameter (convention: T=type, E=element, K=key, V=value).

Type Erasure — What Happens

List and List are both List at runtime. Compiler erases to Object (or bound), inserts casts. Cannot: new T(), instanceof T, new T[], primitives as type params.

Bounded Type Parameters

<T extends Number> — T can only be Number or its subclasses. <T super Integer> — T can be Integer or its superclasses (used for lower bounds, mostly in wildcards).

Wildcards

  • ? extends T (upper bound): I can read T from this collection. Producer.
  • ? super T (lower bound): I can write T into this collection. Consumer.
  • ? (unbounded): Any type. Read-only access (can read Object).
Wildcard Capture Helper

<T> void helper(List<T> list, int i, int j) { T t = list.get(i); list.set(i, list.get(j)); list.set(j, t); } — helper captures the wildcard as named parameter.

PECS Rule (Producer Extends, Consumer Super)

If you get values from a structure, use ? extends. If you put values in, use ? super.

💡 Interview Tip

Common question: "What is type erasure?"

Generics are a compile-time feature. The compiler uses generic type information for type-checking, then erases it — replacing type parameters with Object (or the bound). At runtime, List<String> and List<Integer> are the same List class.

Follow-up: "Explain PECS." — Producer Extends, Consumer Super. If you're extracting values from a collection (producer), use ? extends T. If you're inserting values (consumer), use ? super T.

Wildcards and PECS
// Producer Extends: I read Numbers from the list
void sum(List<? extends Number> nums) {
    double total = 0;
    for (Number n : nums) total += n.doubleValue();
    // nums.add(42);  // NOT allowed — can't write!
}

// Consumer Super: I put Integers into the list
void addOne(List<? super Integer> list) {
    list.add(42);
    // Integer x = list.get(0);  // NOT allowed — must read as Object
}
Key: Generics = compile-time type safety. PECS: extends = read (producer), super = write (consumer).

8. Inner Classes

A class defined inside another class. Used to logically group classes that are only used in one place.

Member Inner Class

A regular class inside another class. Has access to the outer class's members (including private). Requires an outer instance to exist.

Inner Class Compilation

Becomes Outer$Inner.class. Has implicit reference to outer instance. Outer.this.field to disambiguate. Prevents outer from being GC'd if inner is referenced externally.

Static Nested Class

A static class inside another class. Does not have access to the outer class's instance members. Doesn't need an outer instance. Behaves like a top-level class but in a nested namespace.

No Outer Reference

More memory-efficient. Behaves like top-level class in namespace. No access to outer instance fields. Use case: Map.Entry.

Anonymous Class

A class without a name, created inline. Used to instantiate interfaces or abstract classes with a single-use implementation. Often replaced by lambdas (Java 8+).

Local Inner Class

A class defined inside a method. Scope is limited to that method. Can access the method's final/effectively-final local variables.

💡 Interview Tip

Common question: "When would you use a static nested class vs a member inner class?"

Static nested: When the nested class doesn't need access to the outer instance's members. Behaves like a top-level class in a nested namespace. Example: Map.Entry.

Member inner: When you need access to the outer class's private fields. Requires an outer instance. Example: ArrayList.Itr (iterator).

Anonymous: Single-use implementations. Can be replaced by lambdas if the interface is functional.

Anonymous class → Lambda

    @Override
    public void run() {
        System.out.println("Hello");
    }
};

// Lambda (Java 8+) — only if interface is functional
Runnable r2 = () -> System.out.println("Hello");

// Anonymous class without lambda (has multiple methods)
button.addActionListener(new ActionListener() {
    @Override
    public void actionPerformed(ActionEvent e) {
        System.out.println("Clicked!");
    }
});
Key: Member = per-instance. Static nested = per-class. Anonymous = inline single-use. Lambdas replace many anonymous classes.

9. Java 8 Features

The biggest language change in Java history — lambdas, streams, functional interfaces, Optional.

Lambda Expressions

Concise syntax for implementing functional interfaces. (parameters) -> expression or (parameters) -> { statements; }. The compiler infers types and creates the implementation.

Lambda Scope Rules

Lambdas cannot shadow enclosing variables (no new scope for this). Captured variables must be effectively final (not reassigned after initialization). this inside a lambda refers to the enclosing object, not the lambda instance (unlike anonymous classes).

Key Functional Interfaces (java.util.function)

  • Predicate<T>: boolean test(T t) — filter.
  • Function<T,R>: R apply(T t) — transform.
  • Consumer<T>: void accept(T t) — consume (side effect).
  • Supplier<T>: T get() — provide.
Primitive Specializations

IntFunction, IntPredicate, IntConsumer, IntSupplier, LongFunction, DoubleFunction. Using primitives avoids autoboxing in hot paths.

Method References

Shorthand for lambdas that just call an existing method: Class::staticMethod, instance::method, Class::instanceMethod, Class::new.

4 Kinds

1) Class::staticMethod 2) instance::method 3) Class::instanceMethod (first arg = receiver) 4) Class::new. Type 3: String::length = (String s) -> s.length().

Optional

A container that may or may not contain a value. Forces you to handle the "no value" case explicitly. Methods: isPresent(), ifPresent(), orElse(), orElseThrow(), map(), flatMap(), filter().

Anti-Patterns

Do NOT: get() directly, use as method parameter, put in collections, use with boxed primitives (use OptionalInt/OptionalLong/OptionalDouble).

💡 Interview Tip

Common question: "What's a lambda expression and how does it work internally?"

A lambda is a concise way to implement a functional interface. Internally, the compiler uses invokedynamic (Java 7+) — the lambda is not compiled to an anonymous class. Instead, a bootstrap method generates the implementation at runtime. This is more efficient than anonymous classes.

Follow-up: "What are the 4 main functional interfaces?" — Predicate<T> (test), Function<T,R> (apply), Consumer<T> (accept), Supplier<T> (get).

Lambda, Method Reference, Optional

// Lambda

// Method reference (even shorter)

// Predicate: filter with lambda

     .forEach(System.out::println);  // Alice

// Optional — never return null from a method!

    return Optional.ofNullable(u);
}

Key: Lambda → functional interface. Method reference → shorter lambda. Optional → no more nulls.

12. Serialization

The process of converting an object into a byte stream (to save to file, send over network, or cache).

Serializable Interface

A marker interface (no methods). If a class implements Serializable, the JVM allows it to be serialized. All fields are serialized by default, including private fields.

serialVersionUID — Why It's Mandatory

If not declared, the JVM computes one from class structure. Adding a field changes the UID → InvalidClassException. Always declare: private static final long serialVersionUID = 1L;. Use serialver tool to generate from compiled classes.

transient Keyword

Mark a field with transient to exclude it from serialization. Useful for sensitive data (passwords) or derived/volatile values (cached calculations).

What Gets Excluded

transient fields skip serialization, reset to defaults on deserialization. Use for: derived fields, sensitive data, non-serializable references.

serialVersionUID

A unique ID that identifies the class version. If the class changes (new fields added, removed), the UID must match. Always declare private static final long serialVersionUID = 1L; to control versioning. Without it, the JVM generates one based on class details — and a tiny change breaks deserialization.

💡 Interview Tip

Common question: "What happens if serialVersionUID changes?"

Deserialization fails with InvalidClassException. The serialVersionUID is used to verify that the sender and receiver of a serialized object have loaded compatible classes. Always declare it explicitly — the JVM-generated one changes with even minor code changes.

Follow-up: "What does transient do?" — Excludes a field from serialization. The field is set to its default value (0, null, false) on deserialization.

Serialization example
class User implements Serializable {
    private static final long serialVersionUID = 1L;

    private String name;
    private transient String password;  // won't be serialized
    private int age;
}

// Serialize
try (ObjectOutputStream oos = new ObjectOutputStream(
        new FileOutputStream("user.ser"))) {
    oos.writeObject(user);
}

// Deserialize
try (ObjectInputStream ois = new ObjectInputStream(
        new FileInputStream("user.ser"))) {
    User loaded = (User) ois.readObject();
}
Key: Serializable = marker interface. transient = skip field. Always declare serialVersionUID.

2. Java Memory Model (JMM)

The JMM defines when one thread's writes become visible to another thread — the formal rules for concurrent visibility.

Core Concepts

  • Visibility: Without synchronization, a thread may never see another thread's writes (due to CPU caches). volatile or synchronized guarantees visibility.
  • Atomicity: Reads/writes to int, boolean, references are atomic. long/double are not guaranteed (unless volatile). i++ is NOT atomic (read + increment + write).
  • Ordering: The compiler and CPU can reorder instructions. volatile and synchronized act as memory barriers that prevent reordering.
Memory Barrier Types

LoadLoad: All loads before complete before any after. StoreStore: All stores before visible before any after. LoadStore: Loads before stores. StoreLoad: Most expensive — full fence.

Happens-Before Rules

The key rule: An unlock on a monitor happens-before every subsequent lock on the same monitor. A write to a volatile field happens-before every subsequent read of that field. Thread.start() happens-before actions in the started thread. Thread.join() happens-after actions in the joined thread.

6 Key Rules

1) Program order. 2) Monitor unlock → lock. 3) Volatile write → read. 4) start() → thread actions. 5) Thread actions → join(). 6) Transitivity.

volatile

Guarantees visibility and ordering. Does NOT guarantee atomicity (compound operations like count++ are still unsafe). Use for flags, not counters.

4. Annotations

Metadata attached to code elements — processed by the compiler or at runtime. The backbone of modern Java frameworks.

Built-in Annotations

  • @Override — compiler checks that the method actually overrides a parent method.
  • @Deprecated — marks code that should not be used (compiler warns).
  • @SuppressWarnings — silences compiler warnings.
  • @FunctionalInterface — ensures the interface has exactly one abstract method.
@Override — Always Use It

Compiler verifies superclass method exists. Without it, a typo creates a new method instead of overriding. Same for @FunctionalInterface.

Custom Annotations

Define with @interface. Can have elements (like methods). Elements can have default values.

Annotation Processing

Processors run at compile time via AbstractProcessor. Can generate files via Filer but CANNOT modify existing code. Used by Lombok, MapStruct, Dagger.

Meta-Annotations (on annotations)

  • @Target: Where the annotation can be applied (TYPE, FIELD, METHOD, PARAMETER, etc.).
  • @Retention: How long the annotation is kept (SOURCE, CLASS, RUNTIME).
  • @Documented: Include in Javadoc.
  • @Inherited: Subclasses inherit this annotation.
@Retention Policy Choices

SOURCE: @Override. CLASS: bytecode only. RUNTIME: via reflection — required for frameworks (Spring, JPA, Jackson).

Common question: "What's the difference between @Retention(SOURCE), (CLASS), and (RUNTIME)?"

SOURCE: Discarded by compiler. Used for code generation/linting (@Override, @SuppressWarnings).

CLASS: Kept in .class file but not available at runtime. Default. Used by bytecode tools.

RUNTIME: Available at runtime via reflection. Required for frameworks (Spring, Hibernate, Jackson).

Non-blocking, buffer-oriented I/O introduced in Java 1.4 and expanded in Java 7 (NIO.2). Designed for high-performance I/O operations.

Core Abstractions

  • Path: File/directory path. Replaces java.io.File. Paths.get("/home/user/file.txt").
  • Files: Utility class with static methods for file operations: Files.readAllLines(), Files.write(), Files.copy(), Files.walk().
  • Channels: Direct connection to I/O sources (files, sockets). Like streams but bidirectional. FileChannel, SocketChannel, ServerSocketChannel.
  • Buffers: Typed containers for data (ByteBuffer, CharBuffer). Position, limit, capacity. flip() to switch from writing to reading.
  • Selectors: Single-threaded management of multiple channels. The foundation of high-performance network servers (Netty, Vert.x).
Channel + Buffer + Selector

Channel: Bidirectional (replaces streams). Buffer: Typed container (ByteBuffer). Selector: Single-threaded multiplexer for non-blocking I/O.

💡 Interview Tip

Common question: "What's the difference between Java I/O and NIO?"

I/O (java.io): Stream-oriented, blocking, one direction at a time. NIO (java.nio): Buffer-oriented, can be non-blocking, bidirectional (channels). NIO also provides Selectors — single-threaded management of multiple channels for high-performance networking.

Practical use: Files.lines() and Files.write() from NIO are the cleanest file I/O APIs in modern Java.

NIO — Files API
// Read all lines (lazy stream)
try (Stream<String> lines = Files.lines(Paths.get("data.txt"))) {
    lines.forEach(System.out::println);
}

// Write to file
Files.write(Paths.get("out.txt"), List.of("line1", "line2"));

// Walk directory tree
try (Stream<Path> paths = Files.walk(Paths.get("/home"))) {
    paths.filter(Files::isRegularFile)
         .forEach(System.out::println);
}

// FileChannel example
try (FileChannel channel = FileChannel.open(
        Paths.get("data.bin"), StandardOpenOption.READ)) {
    ByteBuffer buf = ByteBuffer.allocate(1024);
    channel.read(buf);
    buf.flip();  // prepare to read
    while (buf.hasRemaining()) System.out.print((char) buf.get());
}
Key: Path + Files = modern file I/O. Channels = fast I/O. Selectors = non-blocking network. Use Files.lines() for text.

7. Functional Programming

A programming paradigm that treats computation as the evaluation of mathematical functions — avoiding mutable state and side effects.

Lambda Internals

Lambdas are compiled using invokedynamic (Java 7+). The JVM generates a bootstrap method that links the lambda to a functional interface. No anonymous class is created at compile time — the lambda is lazily created at runtime. This is more efficient than anonymous classes.

invokedynamic

Compiler generates invokedynamic call site. JVM calls LambdaMetafactory at runtime. Lambda = static method called through the functional interface. Cached and reused.

Stream Internals

Streams use lazy evaluation. Intermediate operations are composed into a pipeline. When a terminal operation is called, the pipeline is fused into a single pass over the data. Each element goes through the entire pipeline, not one operation at a time over the whole collection.

Spliterator

Internal iterator powering streams. tryAdvance() — sequential. trySplit() — parallel (fork/join partitioning). Flags: SIZED, ORDERED, DISTINCT, NONNULL, IMMUTABLE, CONCURRENT.

Functional Composition

Combine small functions into larger ones: Predicate.and(), .or(), .negate(). Function.andThen(), .compose().

Currying (Concept)

Transforming a function that takes multiple arguments into a chain of functions that each take one argument. Not natively supported in Java, but achievable with nested lambdas.

💡 Interview Tip

Common question: "How are lambdas different from anonymous classes internally?"

Anonymous classes create a new .class file and generate a new instance each time. Lambdas use invokedynamic — the JVM creates a single implementation at runtime (cached and reused). Lambdas are faster, use less memory, and have less overhead than equivalent anonymous classes.

Stream internals: Intermediate operations are fused into a single pass. Each element goes through the entire pipeline, not one operation at a time over the whole collection.

Functional composition
Predicate<String> startsWithA = s -> s.startsWith("A");
Predicate<String> longerThan3 = s -> s.length() > 3;

// Compose predicates
Predicate<String> valid = startsWithA.and(longerThan3);

List<String> result = names.stream()
    .filter(valid)
    .collect(Collectors.toList());

// Function composition
Function<String, String> trim = String::trim;
Function<String, Integer> parse = Integer::parseInt;
Function<String, Integer> trimThenParse = trim.andThen(parse);

// Currying concept in Java
Function<Integer, Function<Integer, Integer>> add =
    a -> b -> a + b;
int result = add.apply(3).apply(4);  // 7
Key: Lambdas use invokedynamic (efficient). Streams are lazy and fused. Compose with .andThen(), .and(), .or().

8. JVM Internals

The engine that makes Java work — class loading, memory management, execution, and optimization. (See the JVM guide for full details.)

The JVM is covered in depth in the companion guide. Here's the summary of key points for Java Core interviews:

  • Class Loading: Bootstrap → Platform → Application (parent delegation model). Classes are loaded lazily.
  • Memory Areas: Heap (objects) + Stack (method frames) + Metaspace (class metadata) + PC Register + Native Stack.
  • Object Creation: Load class → allocate in Eden (often via TLAB) → zero memory → set header (Mark Word + Klass Pointer) → run constructor.
  • Garbage Collection: Generational (Young: Eden + Survivor → Old). Minor GC is frequent and cheap. Full GC is rare and expensive. G1 is the default since Java 9.
  • JIT Compiler: Tiered compilation — interpreter → C1 (fast) → C2 (aggressive). Inlining is the most important optimization.
  • Metaspace: Replaces PermGen (Java 8+). Uses native memory. Grows dynamically.

9. Design Patterns

Reusable solutions to common software design problems. Not code — templates for solving problems.

Creational Patterns

  • Singleton: One instance per JVM. private constructor + getInstance(). Enum is the safest Singleton.
  • Factory: Creates objects without specifying the exact class. ShapeFactory.create("circle").
  • Builder: Construct complex objects step by step. new PizzaBuilder().size("large").cheese(true).build().
  • Prototype: Clone existing objects instead of creating from scratch. Cloneable + clone().
Singleton — 4 Approaches

1) Eager. 2) Synchronized getInstance. 3) Bill Pugh holder (lazy, no sync). 4) Enum (best — serialization/reflection/thread safe). Always prefer enum.

Structural Patterns

  • Adapter: Convert one interface to another. Arrays.asList() adapts an array to a List.
  • Decorator: Add behavior dynamically. BufferedReader decorates FileReader.
  • Facade: Simplified interface to a complex subsystem. Files.readAllLines() is a facade over NIO.
  • Proxy: Controls access to an object. Spring AOP uses proxies for transactions.
Adapter vs Decorator vs Proxy

Adapter: interface conversion. Decorator: transparently adds behavior. Proxy: controls access (Spring AOP, dynamic proxies).

Behavioral Patterns

  • Strategy: Encapsulate interchangeable algorithms. Comparator is a strategy.
  • Observer: Notify dependents of state changes. Observer/Observable (legacy), or property change listeners.
  • Command: Encapsulate a request as an object. Runnable is a command.
  • Template Method: Define algorithm skeleton, let subclasses fill in steps. Abstract classes with some concrete methods.

10. Collections Internals

How HashMap, ConcurrentHashMap, and TreeMap work under the hood — critical for interviews.

HashMap Internals

Array of buckets (Node<K,V>[] table). Each bucket is a linked list (or Red-Black tree). put(key, value):

  1. Compute hash = hash(key.hashCode()) (additional mixing to reduce collisions).
  2. Find bucket index: (n - 1) & hash (n = table length, always power of 2).
  3. If bucket is empty, place node there.
  4. If collision: traverse the linked list/tree. If key exists, replace value. If not, add new node.
  5. If list length ≥ 8 and table length ≥ 64, convert list to Red-Black tree (Java 8+ optimization).
  6. If load factor exceeded (default 0.75), resize — double table size and rehash all entries.

ConcurrentHashMap Internals

Java 7: Segment-based locking (array of segments, each a mini-HashMap). Lock per segment.
Java 8+: CAS + synchronized per bin. No segments. Better concurrency.

  • put(): CAS to insert in empty bin. Synchronized on the bin's first node if collision.
  • get(): No locking at all (volatile reads).
  • resize(): Multiple threads help with resizing (concurrent resizing).
Java 8+ Redesign

Java 7: 16 segments. Java 8+: CAS + synchronized per bin. Finer granularity. Counting cells for size(). Treeify per bin just like HashMap.

TreeMap Internals

Red-Black tree. Sorted by keys (natural order or custom Comparator). O(log n) for get/put/remove. Self-balancing — no worst-case O(n) like BST.

Fail-Fast vs Fail-Safe

  • Fail-Fast: Detects concurrent modification and throws ConcurrentModificationException. Most non-concurrent collections (ArrayList, HashMap, HashSet).
  • Fail-Safe: Works on a snapshot or copy. No exception. ConcurrentHashMap, CopyOnWriteArrayList.
Key: HashMap = buckets + linked list/tree. Resize at 0.75 load factor. ConcurrentHashMap = CAS + per-bin sync. Fail-fast = exception, fail-safe = snapshot.

11. Performance Optimization

Making Java programs faster — reducing allocations, minimizing GC pressure, and profiling hotspots.

modCount Mechanism

Structural modifications increment modCount. Iterator checks snapshot at creation. Mismatch → ConcurrentModificationException. Best-effort, not reliable for correctness.

Object Pooling

Reuse expensive objects instead of creating new ones. Useful for: database connections (HikariCP), threads (ExecutorService), byte buffers. Not useful for most objects (GC handles short-lived objects efficiently).

Memory Optimization

  • Use primitives instead of wrappers (int[] vs Integer[]).
  • Avoid creating objects in hot loops.
  • Use StringBuilder instead of String concatenation in loops.
  • Be aware of autoboxing in collections — List<Integer> stores objects, not primitives.
  • Use appropriate collection sizes (initial capacity to avoid resize).
Object Header Cost

Every object: mark word (8B) + klass pointer (4B compressed, 8B uncompressed). Array: +4B length. Padding to 8B. Minimal Object: 16B on 64-bit with compressed OOPs.

GC Impact

  • Fewer allocations = less GC work.
  • If objects are short-lived, Young GC handles them cheaply.
  • If objects live too long, they're promoted to Old Gen → Full GC risk.
  • Large heaps increase GC pause times (especially with non-G1 collectors).
Key GC Flags

-Xms/-Xmx — heap size. -XX:+UseG1GC — default. -XX:MaxGCPauseMillis=100 — pause target. -XX:+UseZGC — sub-ms pauses (Java 15+). -XX:+UseStringDeduplication.

Profiling

Don't optimize without measuring. Use: Async Profiler (flame graphs), JFR (low-overhead), JMH (microbenchmarks). The most common mistake is optimizing code that uses 1% of the CPU.

Async Profiler

Uses perf_events + HotSpot agent. Low overhead (<2%), no safepoint bias. Flame graphs. Profiles CPU, allocations, locks. Sampled mode doesn't stop the world.

💡 Interview Tip

Common question: "How does HashMap work internally?"

Array of buckets. put(): hash(key) → bucket index → if empty, place node. If collision: check linked list/tree. Java 8+ converts long lists (≥8) to Red-Black trees for O(log n) lookup. Load factor 0.75 — when exceeded, resize (double + rehash).

Follow-up: "Fail-fast vs fail-safe?" — Fail-fast (ArrayList, HashMap): throws ConcurrentModificationException if modified during iteration. Fail-safe (ConcurrentHashMap, CopyOnWriteArrayList): works on a snapshot, no exception.

💡 Interview Tip

Common question: "What's the biggest performance mistake Java developers make?"

Premature optimization — optimizing code that runs 1% of the time. Always measure first.

Common actual issues: 1) Creating objects in hot loops (GC pressure). 2) Autoboxing in loops (Integer i = 0; i++). 3) String concatenation in loops (use StringBuilder). 4) Not setting initial collection capacity. 5) Using synchronized when Atomic* or Concurrent* would do.

Modern Java has evolved rapidly. These features appear in interviews and are essential for writing idiomatic modern Java.

Module System (Java 9, Project Jigsaw)

module-info.java defines which packages a module exports and which modules it requires. Enforces strong encapsulation at the JVM level. exports com.my.package, requires java.sql. The JDK itself is modularized (java.base is the base module).

Named: JAR with module-info.class. Automatic: plain JAR on module path. Unnamed: classpath. Migration: classpath first, then modules one by one.

Local variables only (not fields, params, returns). Use when right side shows type clearly. Avoid when it obfuscates: var result = someMethod() is bad.

A compact class for immutable data carriers. record Point(int x, int y) {} — automatically generates constructor, equals(), hashCode(), toString(), accessors (x(), not getX()).

Auto-generates: constructor, equals(), hashCode(), toString(), accessors (field-named). All fields private final. Extends java.lang.Record.

Text Blocks (Java 15+)

Multi-line strings with clean formatting. """ ... """. No more escaping newlines or concatenation.

Indentation

Common leading whitespace stripped using closing '''''' position. indent() for control. \s forces space. \ at line end suppresses newline.

Restrict which classes can extend a parent: sealed class Shape permits Circle, Rectangle {}. Subclasses must be final, sealed, or non-sealed.

sealed interface Shape permits Circle, Rectangle { }. Compiler knows all subtypes — exhaustive switch without default. Permits clause lists all direct subclasses.

Lightweight threads managed by the JVM (not the OS). Millions of virtual threads can run on a small number of OS threads. Perfect for I/O-bound workloads. Thread.startVirtualThread(() -> ...) or Executors.newVirtualThreadPerTaskExecutor().

Virtual threads in synchronized blocks are pinned to carrier thread — can't unmount. Use ReentrantLock instead of synchronized in VT-aware code.

13. Garbage Collection Deep Dive

GC is the automatic memory management system of the JVM. Understanding GC is what separates senior engineers from juniors.

Generational Hypothesis

Most objects die young. Based on this observation, the heap is divided into Young Generation (Eden + two Survivor spaces) and Old Generation (Tenured). Objects are allocated in Eden, survive minor GCs via Survivor spaces, and are promoted to Old after many GC cycles. The ratio is tunable (-XX:NewRatio=2 means Old is 2x Young).

Object Allocation Flow

Eden → Minor GC → Survivor 0 (age 1) → next Minor GC → Survivor 1 (age 2) ... → Tenured (age threshold reached). Default promotion age: 15 (6 with parallel GC). Use -XX:+PrintTenuringDistribution to see object ages. Large objects go directly to Tenured (-XX:PretenureSizeThreshold).

GC Algorithms & Collectors

G1 GC (Default since Java 9)

Divides heap into 2048 regions (1-32MB each). G1 is a regionalized generational collector. It identifies regions with the most garbage and collects them first (Garbage-First). -XX:MaxGCPauseMillis=200 (target pause time). G1 does NOT need to collect the entire heap — it collects subsets to meet pause time goals.

ZGC (Java 15+, Concurrent)

Sub-millisecond pause times regardless of heap size. Uses colored pointers and load barriers. Can handle multi-TB heaps. Pauses < 1ms. Best for latency-sensitive applications. -XX:+UseZGC. Virtual memory remapping enables concurrent compaction.

Shenandoah (Java 15+, Concurrent)

Similar pause times to ZGC. Uses brooks pointers (forwarding pointers). Works with compressed OOPs (unlike ZGC initially). -XX:+UseShenandoahGC. Better throughput than ZGC in some benchmarks.

Serial / Parallel

Serial (-XX:+UseSerialGC): Single-threaded, stop-the-world. For small heaps < 100MB. Parallel (-XX:+UseParallelGC): Multi-threaded, still stop-the-world. Best for throughput-oriented batch processing. Default in Java 8.

GC Tuning Flags

-Xms/-Xmx — heap size. -XX:NewRatio — young/old ratio. -XX:SurvivorRatio — Eden/Survivor ratio. -XX:+UseStringDeduplication — dedup identical strings. -XX:+AlwaysPreTouch — pre-commit all pages (no lazy allocation). GC logging (Java 9+): -Xlog:gc*:file=gc.log.

Interview Tip

Common question: "How does G1 GC work?" — Heap is divided into ~2048 regions. G1 identifies the regions with the most garbage (the "garbage-first" regions) and collects them concurrently. It uses remembered sets (RSet) to track cross-region references. Pause time goal: -XX:MaxGCPauseMillis.

Follow-up: "When would you use ZGC instead of G1?" — When your heap is very large (100GB+) and you need sub-millisecond pause times. ZGC's throughput overhead is higher (~10%) compared to G1, but pause times are < 1ms regardless of heap size.

Senior-level question: "What's a Stop-The-World pause?" — All application threads are paused so the GC can perform certain phases (like root scanning, reference processing, compaction). The goal of modern GCs (G1, ZGC, Shenandoah) is to minimize these pauses by doing most work concurrently with application threads.

14. Dynamic Proxies

Dynamic proxies create implementation of interfaces at runtime. They're the foundation of Spring AOP, Hibernate lazy loading, and declarative transactions.

JDK Dynamic Proxy

Uses java.lang.reflect.Proxy and InvocationHandler. Creates a proxy class that implements one or more interfaces at runtime. Every method call on the proxy is forwarded to the handler's invoke() method. Only works for interfaces (not classes).

How It Works Internally

Proxy.newProxyInstance(classLoader, interfaces, handler) — generates a proxy class that implements the given interfaces. The proxy instance is a $ProxyN class (cached). Each method call is dispatched to InvocationHandler.invoke(proxy, method, args). The proxy class is generated as bytecode (not compiled from source).

CGLIB Proxy

Used when you need to proxy a class (not an interface). Creates a subclass of the target class at runtime. Overrides non-final methods. Used by Spring for @Configuration classes and by Hibernate for lazy loading. Limitations: cannot proxy final methods or classes, no-arg constructor needed in the target class.

Spring AOP Example

Spring creates proxy objects around your beans. When you call a @Transactional method, the proxy opens a transaction before the call and commits/rolls back after. Same for @Secured, @Cacheable, etc. Key trap: a method within the same class calling another @Transactional method bypasses the proxy — the transaction annotation is ignored (self-invocation problem).

Interview Tip

Common question: "How does Spring AOP work?" — Spring creates a proxy of your bean at startup. If the bean implements an interface, it uses JDK dynamic proxy. If not, it uses CGLIB. Every method call on the proxy goes through interceptor chain: transaction management, security, caching, and finally the actual method.

Trap question: "What happens if you call a @Transactional method from another method in the same class?" — The transaction is NOT applied. The call goes directly to this (the raw object), not the proxy. Spring can't intercept the call. Fix: inject a self-reference or move the transactional method to a separate bean.

JDK Dynamic Proxy example
// Step 1: Define an interface
public interface Service {
    void doWork(String input);
}

// Step 2: Implement the interface
public class RealService implements Service {
    public void doWork(String input) {
        System.out.println("Working on: " + input);
    }
}

// Step 3: Create an InvocationHandler
public class LoggingHandler implements InvocationHandler {
    private final Object target;

    public LoggingHandler(Object target) {
        this.target = target;
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args)
            throws Throwable {
        System.out.println("Before: " + method.getName());
        Object result = method.invoke(target, args);
        System.out.println("After: " + method.getName());
        return result;
    }
}

// Step 4: Create and use the proxy
Service proxy = (Service) Proxy.newProxyInstance(
    Service.class.getClassLoader(),
    new Class[]{Service.class},
    new LoggingHandler(new RealService())
);

proxy.doWork("test");  // prints Before / Working on: test / After

// Spring @Transactional does exactly this pattern!

15. Java Security & Cryptography

Java provides a comprehensive security framework: cryptography, authentication, public-key infrastructure, and secure communication.

JCA / JCE Architecture

Java Cryptography Architecture (JCA): Provider-based framework. APIs for digital signatures, message digests, certificates, key stores. Java Cryptography Extension (JCE): Encryption/decryption (symmetric + asymmetric), key agreement, MACs. Both use the Provider architecture — pluggable security implementations (SunJCE, BouncyCastle).

Provider Architecture

Security.getProviders() lists registered providers. java -Djava.security.debug=provider for debug. Bouncy Castle is the most popular third-party provider. Providers are ordered by preference — the first provider that supports an algorithm is used. java.security file controls the provider list.

MessageDigest (Hashing)

One-way hash functions: SHA-256 (standard), SHA-512 (stronger), MD5 (broken, do not use). MessageDigest.getInstance("SHA-256").digest(data). Always use a salt with password hashing (SecureRandom generates salt). For passwords specifically: bcrypt, scrypt, or Argon2 (not raw SHA — they're deliberately slow to resist brute force).

Symmetric vs Asymmetric Encryption

Symmetric (AES, DES, ChaCha20): Same key for encrypt and decrypt. Fast, for bulk data. Key size: AES-128/192/256. Mode: GCM (authenticated encryption, preferred), CBC (needs IV), ECB (never use — deterministic). Asymmetric (RSA, ECDSA, EdDSA): Public/private key pair. Slow, for key exchange and digital signatures. RSA key size: 2048-bit minimum.

Hashing vs Encryption — Fundamental Difference

Hashing: One-way (can't reverse). Fixed output size. Used for passwords, data integrity. Encryption: Two-way (can decrypt with key). Variable output size. Used for confidentiality. These are NOT interchangeable. Storing credit cards: encrypt (you need to retrieve). Storing passwords: hash + salt (you should never retrieve the original).

KeyStore & TrustStore

KeyStore: Contains private keys + certificates (your identity). TrustStore: Contains CA certificates you trust (default: $JAVA_HOME/lib/security/cacerts). Used in SSL/TLS. keytool command manages both. keytool -import -alias myserver -keystore truststore.jks -file cert.pem.

Interview Tip

Common question: "Difference between hashing and encryption?" — Hashing is one-way (no key, can't reverse). Encryption is two-way (needs a key, can decrypt). Use hashing for passwords (you verify, never retrieve). Use encryption for data you need to decrypt later.

Follow-up: "Why is ECB mode bad?" — ECB encrypts each block independently — identical plaintext blocks produce identical ciphertext blocks. Patterns in the plaintext are visible in the ciphertext (like the famous "ECB penguin" image). Always use GCM mode.

Java Cryptography examples
import javax.crypto.*;
import javax.crypto.spec.*;
import java.security.*;

// SHA-256 Hashing
public byte[] hashPassword(byte[] password, byte[] salt)
        throws Exception {
    MessageDigest md = MessageDigest.getInstance("SHA-256");
    md.update(salt);
    return md.digest(password);
}

// AES-GCM Encryption (preferred mode)
public byte[] encrypt(byte[] plaintext, SecretKey key)
        throws Exception {
    Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
    byte[] iv = new byte[12];  // 96-bit IV for GCM
    SecureRandom.getInstanceStrong().nextBytes(iv);
    GCMParameterSpec spec = new GCMParameterSpec(128, iv);
    cipher.init(Cipher.ENCRYPT_MODE, key, spec);
    byte[] ciphertext = cipher.doFinal(plaintext);
    // Prepend IV to ciphertext for storage
    ByteBuffer buf = ByteBuffer.allocate(iv.length + ciphertext.length);
    buf.put(iv).put(ciphertext);
    return buf.array();
}

// SecureRandom vs Random
Random weak = new Random();                // predictable (don't use)
SecureRandom strong = SecureRandom.getInstanceStrong();  // true random
Key: JCA/JCE = provider-based crypto. SHA-256 for hashing (with salt). AES-GCM for symmetric encryption. RSA for key exchange. SecureRandom for crypto-random data. Never roll your own crypto.

16. ServiceLoader & SPI Pattern

The Service Provider Interface (SPI) pattern allows defining a service interface with pluggable implementations discovered at runtime.

How ServiceLoader Works

ServiceLoader<MyService> loader = ServiceLoader.load(MyService.class); for (MyService svc : loader) { svc.execute(); }. The loader looks for files in META-INF/services/fully.qualified.InterfaceName containing the fully qualified class names of implementations. This is the mechanism behind JDBC drivers, JPA providers, and many Java extension APIs.

Why SPI Matters

SPI enables plugin architectures and framework extensibility without compile-time dependencies. JDBC: your code references java.sql.Driver (the service interface), but the actual implementation (MySQL driver, PostgreSQL driver) is discovered at runtime via ServiceLoader. Spring Boot auto-configuration uses a similar mechanism (spring.factories).

Common SPI Examples in Java

  • JDBC: java.sql.Driver — database drivers discovered via SPI. No more Class.forName("com.mysql.cj.jdbc.Driver") needed (Java 6+).
  • JPA: javax.persistence.spi.PersistenceProvider — Hibernate, EclipseLink discovered via SPI.
  • JAXP: SAX/DOM parser factories discovered via SPI.
  • Jackson: ObjectMapper discovers modules (JSR310, Hibernate) via SPI.
  • SLF4J: Finds Logback/Log4j2 binding via ServiceLoader.

ServiceLoader vs Dependency Injection

ServiceLoader provides service discovery without DI frameworks. It's built into Java (no Spring/Maven/Gradle needed). However, Spring's DI gives you more control over scopes, lifecycle, and configuration. For simple plugin systems, ServiceLoader is perfect. For complex applications, use Spring/CDI with its component scanning.

Interview Tip

Common question: "How did JDBC drivers load before Java 6?" — Class.forName("com.mysql.jdbc.Driver") — the static initializer of the Driver class registered itself with DriverManager. Since Java 6, the ServiceLoader mechanism handles this automatically via META-INF/services/java.sql.Driver.

Follow-up: "How would you make your library extensible?" — Define a service interface. Document it. Users implement it and add their class name to META-INF/services/your.Interface. Your code uses ServiceLoader.load() to discover all implementations.

ServiceLoader SPI example
// Step 1: Define the service interface
public interface PaymentGateway {
    boolean processPayment(double amount, String currency);
}

// Step 2: Implement it
public class StripeGateway implements PaymentGateway {
    public boolean processPayment(double amount, String currency) {
        System.out.println("Stripe: $" + amount + " " + currency);
        return true;
    }
}

// Step 3: Register in META-INF/services/
// File: META-INF/services/com.example.PaymentGateway
// Content: com.example.impl.StripeGateway
//         com.example.impl.PaypalGateway

// Step 4: Discover and use
ServiceLoader<PaymentGateway> loader =
    ServiceLoader.load(PaymentGateway.class);
for (PaymentGateway gateway : loader) {
    if (gateway.processPayment(99.99, "USD")) {
        System.out.println("Payment succeeded via " + gateway.getClass());
        break;
    }
}
Key: SPI = runtime discovery of implementations. Files in META-INF/services/. Used by JDBC, JPA, Jackson. No compile-time dependency on implementations.
